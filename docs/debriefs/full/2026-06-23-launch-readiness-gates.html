1 · What We Built (and Why It Matters)

This window built the verification scaffolding for the Nuxt → Hugo cutover. The site's content and templates were already done; what was missing was evidence that flipping the switch won't break SEO — that every URL Google has indexed for 18 years still resolves on the new stack. That's exactly what this work proves, and it proves it with scripts we can re-run on cutover day rather than a one-time manual spot-check.

It maps to ROADMAP.md Phase 16 (Deployment) and 07-deployment.md "Netlify Configuration", and every issue rolls up to the Launch Readiness Epic (#150), which is the living cutover runbook. During the same window, Phase 15 (Content Migration) was formally closed — all 16 checkboxes complete.

The five pieces

🗺️
URL parity audit (#174 / #171) — a static diff of the live Nuxt sitemap against the local Hugo build. Proves the right pages exist. Verdict: 347 one-to-one, 26 need a redirect, 0 would 404.
↪️

🖼️

2 · Architecture & Design Decisions

The three-stage parity pipeline

The single most important structural decision was splitting URL parity into three stages with two distinct scripts, rather than one mega-checker. Each stage answers a different question and fails for a different reason.

Why two scripts and not one? audit.py is deterministic and offline — it diffs a committed snapshot of the live Nuxt sitemap against a walk of ./public, so it runs in CI-time with no network and catches a missing page the instant the build changes. verify.py is the opposite: it makes real HTTP requests to the deployed preview, because a page can exist as a file yet still fail to resolve if a Netlify redirect rule misfires or a chain forms. One can't replace the other — they catch different failure classes — so collapsing them would have hidden a category of bug.

The subtle bit: the "cosmetic slash hop" exclusion

This is the decision I'd most want a future maintainer to understand. Netlify automatically 301s a missing trailing slash (/contact → /contact/) — that's the same page, just normalized. Meanwhile the real redirect we care about (/blog/page/1 → /blog/) also gets slash-normalized in the same response. A naïve hop-counter would see two redirects and flag page/1 as a failing multi-hop chain.

So verify.py's classifier counts a hop as a "content hop" only when the path actually changes ignoring slashes:

# docs/url-parity/verify.py — content_hops()
return sum(
    1
    for cur, status, loc in chain
    if status in REDIRECT_CODES and loc
    and cur.rstrip("/") != loc.rstrip("/")   # slash-only hop doesn't count
)

That one comparison is what makes the live buckets (347 / 26 / 0) line up exactly with the static audit's prediction. Without it, the two stages would report different numbers and the whole "proof" falls apart.

Proving the old redirect rule was dead

The PRD shipped a redirect rule for page/1 — but the audit traced it on the live preview and found it never fired:

# What actually happened with the original non-forced rule:
/blog/page/1  ->[301]->  /blog/page/1/  ->[200]  (Hugo alias stub, meta-refresh)

Netlify normalized the slash first, then served Hugo's own page/1/index.html alias file — which shadows a non-forced redirect. The fix needed three things together, and the live gate confirms all three work:

ChangeWhy it's required
force = trueWins over the existing page/1/index.html alias file (a non-forced rule loses to a real file).
trailing-slash fromMatches Netlify's normalized request path, so the rule is reached at all.
:tag placeholderCollapses all 25 tag page/1 URLs into one rule. The live verify (26/26 single-301s) confirms the placeholder matches every tag — the open question from the audit is now answered.

Defense-in-depth noindex for /thank-you

/thank-you is the one genuinely new URL (the Nuxt site processed its contact form via Lambda and had no confirmation page). A form-confirmation page should never appear in search, so it's suppressed two independent ways:

# content/thank-you.md frontmatter
noindex: true          # → robots meta, via seo.html
sitemap:
  disable: true       # → kept out of sitemap.xml

The seo.html gate is {{ if and hugo.IsProduction (not .Params.noindex) }} — production-gated and per-page opt-out. Robots meta tells a crawler that arrives not to index; the sitemap exclusion stops us advertising the page in the first place. Different crawler paths, so both are warranted. verify.py explicitly asserts /thank-you never leaks into the sitemap.

Gates own disjoint slices (the exclude-list philosophy)

Each gate deliberately excludes what another gate owns, so coverage is partitioned rather than overlapping:

🔗
lychee excludes ofreport.com — that's verify.py's job (and checking it here would hit the old live Nuxt site).
🖼️
lychee excludes res.cloudinary.com — covers are probe-covers.rb's job, and bulk automated HEADs to hundreds of derivatives throttle into non-deterministic "connection failed" noise.
📋
Every other exclude entry is documented triage in docs/link-check.md — bot-blocked-but-live (Vimeo, Ford UK), or dead legacy links with no live equivalent — never silent suppression.

The payoff is that each gate is deterministic on its own slice; you never get a red run because two gates fought over the same flaky resource.

POODR spotlight — Single Responsibility & depending on stable things

The audit.py / verify.py split is textbook Single Responsibility: each script has exactly one reason to change. If the build's page set changes, only audit.py is affected; if Netlify's redirect behavior changes, only verify.py cares. Merging them would have given one class two axes of change — the SRP smell.

The second principle on display is depend on something more stable than yourself. probe-covers.rb reads cloudinaryBase straight out of hugo.toml rather than hardcoding the CDN host, and both parity scripts read the committed live-urls.txt inventory rather than re-deriving it. The gates depend on the same source of truth the site does, so they can't quietly drift from it. The one place this isn't achieved — the HERO_TRANSFORMS constant duplicating the partial's preset — is flagged in code with a "MUST stay in sync" comment, i.e. acknowledged coupling rather than hidden coupling.

One clever insight worth keeping

probe-covers.rb probes only one transform per cover (the hero preset), not all six. The reasoning: Cloudinary derivatives are all-or-nothing on the source asset — if the asset exists, every transform of it resolves; if the cover was fabricated/placeholder, every transform 404s. So one representative transform conclusively answers "does this cover resolve?" That's a 6× reduction in network calls justified by how the CDN actually behaves.

3 · Gate Coverage & Quality

This chunk has no unit tests in the traditional sense — it is the test suite for the cutover. The "tests" are four re-runnable gates, each exiting non-zero on failure so a clean run is unambiguous. What I deliberately did not test, and why

Every Cloudinary figure derivative — out of scope by design. Covers have a dedicated probe; bulk HEADs to inline-figure derivatives throttle and would make the gate flaky. Known coverage gap: a typo'd inline figure public ID would not be caught by any current gate. Acceptable for cutover; worth a thought post-launch.
Outbound link liveness as a hard gate — outbound links are triaged, not gated. Many hosts bot-block crawlers (403/429) while working fine in a browser, and some legacy links are simply dead with no replacement. Hard-gating them would mean a perpetually red run for things we can't fix.
Production domain resolution — verify.py runs against the preview today because production is still the Nuxt site. The script takes a base-URL argument precisely so it becomes the cutover-day check against ofreport.com.
Thin spot to watch The gates are powerful but manual — none are wired into the Netlify build or CI yet. For a one-time cutover that's the right call (you run them deliberately, in order, before flipping). Post-launch, the lack of automation is a regression risk: a future content edit could reintroduce a broken internal link and nothing would catch it until the next manual run.
