1 · What We Built (and Why It Matters)

Phase 6 delivers the four custom Hugo shortcodes that article content is built from — the reusable components authors (and the Phase 15 migration script) drop into markdown. This is the layer that sits on top of the page scaffolding from Phases 1–5: the site already has a base layout, blog listing, and single-article template, so these shortcodes are the pieces that go inside an article.

All four are implemented, verified, and merged:

2 · Architecture & Design Decisions

The four were built simplest-first (button → svg → callout → figure) so each introduced one new Hugo concept and the last could compose the rest. They share a deliberate house pattern.

The shared shortcode pattern

Every shortcode reads named params with .Get, fails the build loudly on a missing required param via errorf (with .Position for a file:line pointer), and emits Tailwind-styled markup. Establishing this in button.html first meant the next three were variations on a known theme rather than four bespoke designs.

ShortcodeNotable decision
buttonHugo parses unquoted external=true as a bool, not a string — normalized with printf "%v" so both external=true and external="true" work. External links also get an icon plus a visually-hidden "(opens in a new tab)" note for screen readers.
svgTwo distinct failure modes: missing name param → errorf (author error, fail the build); missing file → warnf (Phase 8 logos legitimately don't exist yet, so don't break the build). Inlines via .Content | safeHTML so the SVG is CSS-styleable.
calloutFirst paired shortcode. .Inner is raw, rendered with .Page.RenderString (dict "display" "block") — block display is the key detail (the default is inline, which wouldn't wrap paragraphs). The [&>:first-child]:mt-0 utilities reset prose margins so the box padding stays even.
figureTwo sized URLs from one src; not-prose to match the cover figure; centered caption; lightbox-compatible markup. See the data flow below.

The architecturally significant call: where public-ID handling lives

The figure shortcode accepts src as either a full Cloudinary URL or a bare public ID. My first cut detected the bare ID and built the full URL in the shortcode (hardcoding Cloudinary's /image/upload/ path), then handed off to the cloudinary-url.html partial to apply the size transform. The /simplify pass flagged this, and I moved the expansion into the partial.

Two reasons it belongs there: (1) cloudinary-url.html is the single owner of Cloudinary URL construction — the shortcode shouldn't know Cloudinary's URL format; and (2) the partial previously silently no-op'd on a bare public ID (its replace "/upload/" found nothing and returned the ID unchanged — a broken URL with no warning). Centralizing the expansion fixes that footgun for every caller. The change is backward-compatible: full https:// URLs hit hasPrefix "http" and skip the new branch, so the two existing cover callers (single.html, article-card.html) render byte-identical URLs.

POODR · Single Responsibility

cloudinary-url.html has one job: turn a Cloudinary src + preset into an optimized URL. The first cut of figure smeared that job across two files — the shortcode knew Cloudinary's /image/upload/ structure and built part of the URL, then the partial finished it. Moving the public-ID expansion into the partial restores the single responsibility: the partial owns every step of "construct a Cloudinary URL," and figure depends only on the abstraction ("give me a URL for this src and preset"), not on Cloudinary's URL format. The payoff is concrete — any future caller that passes a bare public ID is correct for free, and the silent-no-op footgun is gone. This is SRP and dependency-direction working together: the high-level shortcode no longer depends on a low-level URL detail.

Sizing — the one PRD deviation

The issue note for figure said the display image should use the existing article preset. But article is width-uncapped (it serves the full original), which both contradicts the established ~768px inline-body-image convention and makes "a larger lightbox" impossible. After confirming the convention against the current Nuxt blog, I added a capped figure preset (w_768) for the inline image and a lightbox preset (w_1600) for the zoom — the correct click-to-zoom pattern. The cover image (separate, in single.html) still uses article and is unaffected. This was logged in CHANGELOG.md and has since been folded into 05-shortcodes.md §"Figure/Image Shortcode".

Alternatives considered and rejected

🔁
markdownify for callout inner content → chose .Page.RenderString: it respects site render hooks and renders in the page's context (so relative links resolve).
🎨
Override prose styles on the figure → chose not-prose: cleaner and matches the cover figure exactly. Note the principled split — callout keeps prose (it renders a markdown body), figure opts out (it's a structural image).
🧭
Keep public-ID logic in the shortcode (the "rule of three" reading) → chose to centralize: this completes an existing helper's contract rather than minting a new abstraction, so the rule-of-three guard doesn't apply.
3 · Test Coverage & Quality

This is a Hugo static site — there's no unit-test suite. The meaningful checks are the production build (does every template compile and render?) and manual inspection of rendered HTML (does each shortcode emit the right markup across its variants?). My approach this phase: for each shortcode, drop a temporary draft article exercising every variant, build, inspect the generated HTML, then delete the draft. Sample content was folded into each issue's acceptance criteria rather than committed as a permanent fixture.

Caught by verification

The button external variant first rendered without target/rel: Hugo handed back a bool where the code compared against the string "true". Only inspecting the rendered HTML surfaced it — fixed before the PR. It's the reason the verify-by-reading-output step earns its keep.
